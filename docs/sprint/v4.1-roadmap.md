# Aiden v4.1 — backlog (rolling)

Items banked from Phase 16–21 reviews. Ordered by source, not priority — priority gets assigned at v4.1 sprint kickoff.

## From Hermes architecture retrospective (Phase 21, 2026-05-06)

- **Trace schema standardization.** Declare honesty-trace + tool-call-trace schemas explicitly (TypeScript interfaces alone aren't enough — need a JSON Schema or zod schema with versioning). Today the trace shape is implicit in `moat/honestyEnforcement.ts::HonestyTraceEntry`.
- **OS-level plugin sandboxing.** Phase 17 deferred OS sandbox. v4.1 candidates: Node `vm.Context` (weak), Deno-style permissions (rewrite cost), Linux seccomp via a sidecar process (best parity with Hermes's deferred plan).
- **Unified policy engine.** Today: `moat/approvalEngine.ts` (gating writes) + `moat/dangerousPatterns.ts` (shell classification) + `moat/honestyEnforcement.ts` (post-hoc claims) + `moat/ssrfProtection.ts` + `moat/tirithScanner.ts`. Each has its own rule set. v4.1: single `policy.yaml` consulted by all five.
- **Skill activation telemetry.** Today no measurement of which skills the model actually views/calls. v4.1: emit `skill_view` events to `<aiden-home>/logs/skill-telemetry.jsonl`; surface in `/skills stats`.
- **Per-provider streaming strategy.** Phase 16c buffer-and-suppress is uniform. Hermes hybrid: stream when reliable provider, buffer when flaky. Wire `streamingPolicy: 'stream' | 'buffer' | 'auto'` per registry entry.

## Phase 16d revisit — memory snapshot lifecycle

- **Frozen snapshot vs invalidate-on-write.** Phase 16d chose invalidate-on-write so the next turn reflects `✓ Saved`. Hermes warns this causes instruction drift on long sessions. v4.1: evaluate freeze-+-`/refresh` UX; gate behind a config toggle.

## Phase 18 / 21 OAuth gaps

- **Live `/codex/models` probe.** Today's hardcoded model list (Phase 21 #6a) was wrong once already; bound to drift again. Port Hermes `agent/model_metadata.py:_fetch_codex_oauth_context_lengths` (1 h cache, fallback to hardcode).
- **Reasoning config support.** Codex Responses API accepts `reasoning: {effort, summary}` and `include: ['reasoning.encrypted_content']`. Wire as adapter input, mapped from a `reasoning_effort` config key.
- **session_id + prompt_cache_key.** Phase 21 #6b banked these. Pass session-id from `chatSession` through to `CodexResponsesAdapter` so cross-turn caching works on the Codex backend.
- **OS keychain for tokens.** Today `tokenStore` uses machine-bound AES (obfuscation, not protection). v4.1: bind to Windows DPAPI, macOS Keychain, libsecret on Linux — actual secret-boundary protection.
